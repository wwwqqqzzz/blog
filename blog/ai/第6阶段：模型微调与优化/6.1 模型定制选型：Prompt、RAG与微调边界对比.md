# 6.1 业务定制指南：Prompt、RAG与微调的三岔路口

:::info 我们为什么需要让模型“变聪明”？
从第 1 到 5 阶段，我们学习了如何向基础大模型发问（Prompt）、如何让它查资料（RAG）、甚至如何让它变成能够自主使用工具的智能体（Agent）。
这些操作本质上都是在一个前人已经预训练极其完备好的通用聊天大模型基础上，通过前置和外挂手段去扩充能力。

但是，通用模型并不懂每家企业的“黑话”、特定行业的代码规范，也无法在特定情绪输出上达到完美的拟人感。当 Prompt 变得臃肿、RAG 的召回不够准确时，我们需要走向底层——即修改模型本身的“大脑神经元”。
本章，我们将清晰地梳理：在面对特定的定制化需求时，何时使用 Prompt，何时使用 RAG，何时必须进行微调 (Fine-Tuning)。
:::

---

## 1. 成本最低的防线：提示工程 (Prompt Engineering)

我们在第 3 阶段已经深入探讨过，这是最基础的干预手段。其本质是通过精心设计的语境与指令，激活大模型在预训练海量语料中已经记住但未能迅速调取的隐含特征。

- **优势**：零研发成本，立竿见影，随改随用；不需要任何显卡运算去改变权重。
- **局限**：对严重缺乏的私有知识无效。例如，如果你的模型在预训练时从未见过“2025款某品牌内部跑车底盘测试数据”，即便采用最优秀的 Prompt，也只能得到凭空捏造的幻觉（Hallucination）。
- **最佳场景**：快速测试验证 (Proof of Concept)；针对基础语言转化、公有领域的知识总结，以及改变文本情感风格（如要求“使用法务公文的严谨语言”）。

## 2. 外部的记忆数据库：检索增强生成 (RAG)

当模型需要大量私有知识时，就到了第 4 阶段中提到的 RAG。其中心思想是外挂知识库：遇到不懂的问题，不是去逼迫模型回答，而是去检索外挂的文档片段，再让模型基于片段做推理作答。

- **优势**：知识更新极快，删减、增添文件极其便利且权限控制清晰（谁提问就只能检索属于他的文件域）。不需要对大模型底层神经元进行改造。
- **局限**：非常依赖向量数据库召回的精度；在需要理解长文逻辑连贯性和跨篇章发散推演时，容易因检索到的片段碎片化而导致回答局限。
- **最佳场景**：企业内部知识库问答系统、客服智能回复、具备时效性的新闻财报助手。

## 3. 直接动刀修改神经元：模型微调 (Fine-Tuning)

上述两者始终未曾动过大模型底部的矩阵权重参数。而微调则是将大模型“开颅”，利用特定的新领域标注数据，通过计算梯度并反向传播，对网络里高达数十亿参数的计算偏差进行校正，强行扭转它的潜在认知地图。

### 3.1 什么是微调？

微调（Fine-tuning）并不像预训练那样需要万张显卡狂跑数个月。它是基于一个已经具备通用语言能力的开箱即用模型（例如 Llama-3 8B Base 或 Qwen 7B Base），在较小的高质量领域特定数据集（例如包含几万条代码书写范例的文件集）上，进行的二次专门训练。

### 3.2 什么时候必须进行微调？

当面对下列强特定场景，且 Prompt 和 RAG 已无法解决时，微调是唯一选择：

1. **特定的输出对齐与高强度结构化**：你希望模型无论在何种情况，都能在不给予很长 Prompt 的前提下，自发地采用极度严格的 JSON 数据树结构或者专有的查询前缀标记进行回复。
2. **特定的行业暗语与极度垂直的发音风格**：你需要一个语气像古代武侠长者，满口诗文且具备特定价值观的 NPC 角色，甚至要他理解武侠圈子的隐藏梗。通用模型经过大量安全对齐，很难用纯 Prompt 保持这种风格长期稳定地一致输出。微调能从内部深层扭转模型的输出分布函数。

### 3.3 劣势与成本

- 需要采集和整理人工标注的高质量微调数据集。
- 尽管存在高效微调方法（如 LoRA），其所需耗费的显存资源和知识理解门槛，较之前两者依然有了巨大的爬升。微调一个大模型可能需要数小时乃至数天的显卡时间计算。
